# 飞书机器人接入 - Codex/Claude Code 执行提示词包

> 用法：复制任意一组提示词给 Codex 或 Claude Code。
> 
> 要求：每组都强制模型先读取仓库脚本与日志，再给结论，不允许凭空猜。

---

## 使用总规则（每次都加在提示词最前面）

```text
你现在在仓库根目录执行。先读当前实现再给结论：
1) 必读文件：
- 06-工具/scripts/feishu_ingest_server.py
- 06-工具/scripts/feishu_ingest_router.py
- 06-工具/scripts/link_to_quotes.py
- 06-工具/scripts/run-feishu-ingest.ps1
- 06-工具/scripts/run-feishu-tunnel.ps1
- 06-工具/scripts/.env.feishu.example
2) 必查日志/产物：
- 06-工具/data/feishu-ingest/state.db（若可读）
- 03-素材库/金句库/导入记录/YYYY-MM-DD-feishu-import.md
- 03-素材库/对标链接库/YYYY-MM-DD-feishu-links.md
3) 先输出“已核对的事实”，再给诊断和操作步骤。
4) 不要让我手工猜路径，命令必须可复制执行（PowerShell）。
```

## 1) 环境预检

### 输入模板
```text
[任务] 做飞书接入预检，不改业务逻辑。
[要求]
- 检查 Python / cloudflared / 依赖 / 环境变量是否满足运行条件。
- 执行 run-feishu-ingest.ps1 -CheckOnly 和 run-feishu-tunnel.ps1 -CheckOnly。
- 给出“通过/失败”表格和最短修复命令。
```

### 预期输出
- 一张检查表：项目、结果、证据。
- 一组可直接执行的修复命令。

### 失败时追加提示
```text
如果你无法判断，请直接执行命令拿证据，不要让我口头补充。
```

---

## 2) 飞书权限核查

### 输入模板
```text
[任务] 核对飞书应用权限与事件订阅是否符合当前仓库实现。
[我当前已开权限]
- im:message
- im:message.group_at_msg:readonly
- im:message.p2p_msg:readonly
- im:message:send_as_bot
[我当前事件]
- im.message.receive_v1
[要求]
- 对照代码逐项核对是否足够。
- 指出缺失项和风险项。
- 给我“飞书后台逐点击路径”。
```

### 预期输出
- “必须有/建议有/可不配”三段清单。
- 一步一步后台点击路径。

### 失败时追加提示
```text
如果你不确定某个权限，请在结论中写“需复核”并给验证命令或日志判断法。
```

## 3) 回调连通诊断

### 输入模板
```text
[任务] 诊断为什么飞书事件回调失败。
[现象]
- 回调地址：<粘贴你的地址>
- 飞书日志：httpCode=<粘贴>
- 错误信息：<粘贴>
[要求]
- 按网络层、接口层、鉴权层三层排查。
- 每层给“如何验证”和“如何修复”。
- 输出按优先级排序。
```

### 预期输出
- 三层排查树。
- 每个节点都有命令与预期返回。

### 失败时追加提示
```text
请你直接给“最小可用链路”方案（本机+TryCloudflare）并说明如何回滚。
```

## 4) 403 专项修复（token / timestamp / signature）

### 输入模板
```text
[任务] 我遇到 403，帮我做定点修复。
[错误详情]
<粘贴飞书事件日志 json>
[要求]
- 明确是 invalid token 还是 invalid timestamp/signature。
- 给出最短修复步骤（不超过 10 步）。
- 给修复后验证命令。
```

### 预期输出
- 错因判定 + 证据。
- 10 步内可执行修复流程。

### 失败时追加提示
```text
如果结论依赖时间同步，请把“校准系统时间”的具体命令写出来。
```

---

## 5) 400 errCode=99992354（open_message_id invalid）专项修复

### 输入模板
```text
[任务] 修复飞书回执失败：open_message_id invalid（99992354）。
[日志]
<粘贴服务端日志和飞书服务端 API 日志>
[要求]
- 判断是“仅回执失败”还是“回调整体失败”。
- 检查 message_id 来源链路。
- 给出可直接执行的修复和验证步骤。
```

### 预期输出
- 影响面判断（入库是否受影响）。
- message_id 追踪路径。
- 修复后复测步骤。

### 失败时追加提示
```text
如果暂时无法修复回执，请给“先保证入库成功”的临时兜底配置。
```

## 6) 链接抓取失败排查

### 输入模板
```text
[任务] 我发链接后没有提取正文，帮我排查。
[样本链接]
<粘贴链接>
[要求]
- 检查 link_mode 路由是否命中。
- 检查 url-reader 调用链和输出目录。
- 检查最终落库文件是否只保留“标题/正文/原文链接”。
- 给修复建议，不改变“链接不进金句库”规则。
```

### 预期输出
- 路由 -> 抓取 -> 清洗 -> 写入 四段证据。
- 失败点对应修复动作。

### 失败时追加提示
```text
请附上“用同一链接复现问题”的完整命令和期望输出。
```

## 7) 幂等验证

### 输入模板
```text
[任务] 验证同一 event_id 不会重复入库。
[要求]
- 读取 state.db 或调用 replay 接口验证。
- 给出一次“首次处理”和一次“重复处理”结果对比。
- 标注哪些文件应当不再变化。
```

### 预期输出
- 首次 vs 重放对比表。
- 幂等结论。

### 失败时追加提示
```text
如果 state.db 不可直接读，请改用接口与日志交叉验证并写清限制。
```

## 8) 发布前最终检查

### 输入模板
```text
[任务] 在我切换到长期运行前，给我一份发布前最终检查。
[要求]
- 覆盖：权限、事件、回调、环境变量、服务健康、文本验收、链接验收、幂等。
- 输出 checklist（可打勾）。
- 每项后面都给“失败时修复入口”。
```

### 预期输出
- 一份完整 checklist。
- 每项都能落到具体命令/页面。

### 失败时追加提示
```text
请把检查项按“阻塞发布 / 可带病上线”分级。
```

## Codex 与 Claude Code 的差异提示

1. 两者共用提示词主体。
2. 如果是 Codex：补一句“优先直接执行命令给证据”。
3. 如果是 Claude Code：补一句“先读文件再执行，避免漏读仓库规则”。
4. 两者都要禁止“纯口头推测”，必须给证据链。
