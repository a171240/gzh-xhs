# runtime artifact 治理说明

## 目的
本文件用于定义“哪些内容属于运行产物，应该如何处理”，避免后续整理仓库时把人工资产、方法论文档或可复用内容误判成垃圾产物。

## 当前确认的 runtime artifact 范围

### 1. 根目录 `reports/`
典型内容：
- 发布/排版/抓取截图
- smoke test 结果
- dry-run payload
- 临时 preview、render manifest、副本日志

处理原则：
- 允许随运行持续增长
- 默认不视为 source，也不视为长期内容资产
- 需要保留证据时，优先保留最近一次可复现样本，而不是累计保存所有历史试跑

### 2. `06-工具/data/`
当前已观察到的子目录：
- `automation/`
- `codex-commander/`
- `feishu-ingest/`
- `feishu-orchestrator/`
- `ingest-writer/`
- `profiles/`

处理原则：
- 全部按 runtime state 对待，不纳入源码真相源
- 其中 `profiles/` 虽然也是 runtime state，但承载持久登录态，不能在“整理仓库”时顺手删除
- 日志、状态文件、心跳、队列副本可以按运维需要轮换或清理，但不得影响仍在使用的进程

### 3. 本地依赖与缓存
- `node_modules/`
- `__pycache__/`
- `*.pyc`
- 本地 npm / Python 临时缓存

处理原则：
- 一律不纳入 source
- 可按锁文件或 requirements 重建

## 明确不属于 runtime artifact 的内容
以下内容即使名字里带“日报 / 报告 / 生成 / 分析”，默认仍视为长期内容资产：
- `01-选题管理/`
- `02-内容生产/` 下人工维护的模板、方法论、正式内容
- `03-素材库/`
- `04-数据与方法论/`
- `05-业务运营/`

典型例外说明：
- `03-素材库/对标链接库/分析报告/` 默认按内容资产处理
- `04-数据与方法论/方法论沉淀/日报/` 默认按内容资产处理
- 只有能明确证明是纯运行产物的文件，才允许单独移出或清理

## 清理原则
- 先判断所有权，再决定是否清理
- 先确认是否仍被运行链路依赖，再做删除或归档
- 不做目录级一刀切删除
- 不为了“让仓库看起来干净”去误删人工经营资产

## 建议操作
- 需要保留调试证据时：保留最近一次成功/失败样本即可
- 需要清理运行痕迹时：优先清理 `reports/` 和无用日志，而不是 `01-05`
- 需要迁移长期可复用内容时：移到 `01-05` 对应资产区，不留在 runtime 目录

## 与 `.gitignore` 的关系
- `.gitignore` 只负责阻止 runtime artifact 继续污染 source 视图
- `.gitignore` 不是“自动判定内容价值”的工具
- 因此 `.gitignore` 规则不得扩张到 `01-05` 的长期资产目录
