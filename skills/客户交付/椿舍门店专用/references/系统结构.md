# 椿舍 Skill 系统结构

## 1. 这是什么

`椿舍门店专用` 只有一个 skill 入口：

```text
椿舍内容：关键词=...
```

复杂的不是 skill 数量，而是这个 skill 内部带了一条内容生成 workflow。

## 2. 现在的真实层级

### 2.1 入口层

- 用户只看见一个 skill
- 负责接收 `关键词 / 输出 / 角色 / 模式 / 金句主题`

### 2.1.1 目录结构

- `SKILL.md`：入口、协议、读取顺序、最小工作流
- `references/`：方法论、业务事实、审稿与口语规则
- `assets/`：本地运行素材，如 `专用选题池.json`、`关键词包.xlsx`
- `archive/`：历史版本提示词和旧技能稿，只作追溯

### 2.2 本地规则层

这层不走模型，直接在本地完成：

- brief 解析
- 关键词路由
- 选题池读取
- 最近历史去重
- 题目挑选
- source pack 构建
- 金句候选筛选
- 置顶评论模板
- 回复模板

### 2.3 模型层

现在默认只保留两个模型节点：

#### `draft_package`

负责：

- 定人
- 定冲突
- 定买点
- 出 3 个标题
- 出正文首稿

不再负责：

- 置顶评论
- 回复模板
- 解释整套策略

#### `polish_package`

负责：

- 修跳句
- 修缺桥
- 去解释味
- 做口语化
- 结尾可选轻放 0-1 句金句

不再负责：

- 重新定题
- 重新定规矩
- 重写整篇策略

## 3. 三种模式

### 3.1 快速

链路：

`本地选题 -> draft_package -> 本地校验通过则直接落稿`

特点：

- 最快
- 默认不再跑润稿
- 适合高频出稿

### 3.2 平衡

链路：

`本地选题 -> draft_package -> polish_package -> 落稿`

特点：

- 质量更稳
- 速度比快速慢
- 现在仍是主力双调用链

### 3.3 严格

链路：

`本地选题 -> draft_package -> polish_package -> 必要时补修`

特点：

- 最慢
- 只在明确要求时使用

## 4. 为什么以前会显得复杂

以前这条线里，很多本该本地完成的事情，也交给模型做了：

- `source_pack`
- 置顶评论
- 回复模板
- 金句注入判断
- 审稿放行

结果就变成：

- 一条文案要跑很多节点
- 每个节点都像在重读一遍方法论
- 速度慢
- 出问题也难排查

## 5. 现在已经砍掉的复杂度

- `source_pack` 已经改成本地构建
- `置顶评论 / 回复模板` 已经改成本地生成
- `快速模式` 允许跳过 `polish_package`
- `chunshe_wj` 不再在运行前把整包上下文文件读给模型

## 6. 现在还保留的必要复杂度

这些仍然保留，因为它们直接影响内容质量：

- 关键词意图路由
- 选题去重
- 正文首稿生成
- 连贯与口语化修稿
- 高风险词边界拦截

## 7. 当前最推荐的理解方式

不要把椿舍看成“一个大 prompt”。

更准确的理解是：

```text
一个 skill 入口
    -> 一层本地规则系统
    -> 一层最小必要的模型生成
```

这样它既还是一个 skill，又不会再被误解成“只是换个 prompt 写文案”。
